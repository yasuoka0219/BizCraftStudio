あなたは熟練のフルスタックエンジニア兼Webデザイナーです。
BizCraft Studioの既存サイトに「お客様の声（Testimonials）」ページ/セクション/カードUIが既に存在します。
今回は “新規追加” ではなく、既存の「お客様の声」カード（またはデータ）を、下記の1事例の内容に「差し替え」してください。

# 目的
既存のTestimonials（お客様の声）の表示構造・デザイン・CSS・コンポーネントは一切崩さず、
表示されるテキスト内容のみを、指定の事例に差し替える。

# 絶対条件（重要）
- レイアウト/デザイン/コンポーネント構造は維持（UIはそのまま）
- 既存のCSS/クラス名/余白/フォント/色/角丸/影などを変更しない
- 既存のカードが複数ある場合：
  - まず「どのファイル/どのデータ構造でTestimonialsが管理されているか」を特定する
  - 既存カードのうち「先頭の1件」または「指定ID/slugの1件」をこの事例に置換する
  - 既存の件数や並びは変えない（必要がなければ）
- 文言は改変しない（改行や句読点の微調整のみOK）
- 可能なら既存の多言語や型定義（TypeScript等）に合わせ、型エラーを出さない
- 変更箇所が分かるようにコメントを付与（該当ファイルに最小限）

# 作業手順（やること）
1) コードベース内で Testimonials / voice / review / testimonial / customers などで検索し、
   お客様の声の「データ定義」または「カードに流し込むprops」を管理している場所を特定する
   例：/data/testimonials.ts, /constants/testimonials.js, CMSのJSON, markdown, Notion連携など

2) 既存のカード表示ロジックはそのままに、該当データ（1件）を下記内容に差し替える
   - もしカードの項目が固定（例：title, company, industry, employeeCount, problem, reason, actions, results, message）の場合は、その形にマッピングして入れる
   - もし本文が1つの長文フィールドしかない場合は、下の文章をセクション見出し付きで整形して入れる
   - もし箇条書きが配列になっている場合は、actions/results を配列で入れる

3) 実装後、プレビューで既存UIが崩れていないことを確認する（レスポンシブ含む）

# 差し替える「お客様の声」コンテンツ（このまま使用）
【事例タイトル】
“何をしている会社か”が一瞬で伝わり、問い合わせの質が変わりました

【属性】
業種：ITサービス（中小企業）
従業員数：12名

【導入前の悩み】
以前のホームページは見た目こそ整っていましたが、
「結局この会社は何が強みなのか」「誰向けのサービスなのか」が分かりづらく、
問い合わせも月に1〜2件あるかどうかという状態でした。
営業時にも毎回ゼロから説明が必要で、正直“HPが営業の役に立っていない”と感じていました。

【BizCraft Studioを選んだ理由】
単なるデザイン制作ではなく、
「誰に、何を、どう伝えるか」を最初に一緒に整理してくれる点に惹かれました。
ヒアリングの中で自社の強みや価値を言語化してくれたことで、
これまで自分たちでも曖昧だった部分がクリアになったのが印象的でした。

【実施した内容】（箇条書き）
- ターゲット顧客の再定義
- 自社サービスの強み・価値の言語化
- サイト全体の構成と導線の再設計
- 問い合わせにつながるコンテンツの整理・改善
「情報を増やす」のではなく、「伝える順番を整える」設計をしてもらいました。

【得られた成果・変化】（箇条書き）
- 問い合わせ数が約2.5倍に増加
- 初回商談時の説明時間が短縮
- 相談内容の精度が上がり、ミスマッチが減少
「HPを見て内容を理解した上で連絡しました」と言われることが増え、
営業ツールとして明らかに機能するようになりました。

【これから検討する方へ】
「デザインはきれいだけど成果が出ない」と感じている方には、
一度BizCraft Studioに相談する価値があると思います。
見た目だけでなく、“ビジネスの成果”を本気で考えたサイトを作ってくれます。

# 形式の注意（カード項目に合わせる）
- 既存が「短文コメント＋氏名/会社名」形式しかない場合：
  - title（見出し）＝上記タイトル
  - comment（本文）＝「導入前の悩み」〜「これから検討する方へ」を1本にまとめて入れる
  - name/company が必須なら、架空で自然な表記にする（例：株式会社◯◯ 代表 / ◯◯様）
  ※ただしUIの項目数・表示枠は増やさない

# 納品要件（必ず出す）
- 変更したファイル名とパス
- どのデータ（何件目/どのID）を差し替えたか
- 既存UIが崩れていない確認結果（簡単でOK）
- 可能なら差分が追えるように最小変更でコミット単位の変更にする
