あなたは熟練のフルスタックエンジニア兼Webデザイナーです。
現在のWebサイトには「お客様の声（Testimonials）」のページ/セクション/カードUIが既に存在します。
今回は既存のカードを差し替えるのではなく、既存カードのFMT（データ構造・UI・デザイン）を一切崩さずに、
下記の【AI記事制作（SEO）】事例カードを“新規で1件追加”してください。

# 目的
既存Testimonialsの表示構造・デザイン・CSS・コンポーネントは変更せず、
表示されるカードを1件増やす（既存の並び・UIのまま追加）。

# 絶対条件（重要）
- レイアウト/デザイン/コンポーネント構造は維持（UIは一切変更しない）
- 既存のCSS/クラス名/余白/フォント/色/角丸/影などを変更しない
- 既存カードのFMT（フィールド名・型・構造）を厳密に踏襲して1件追加する
  - 例：title, name, company, role, comment, tags, avatar, rating 等
  - 配列やオブジェクト構造がある場合も同一形式で追加
- TypeScript等の型定義があれば必ず合わせ、型エラーを出さない
- 既存カードの順番や件数（既存分）は変えない
  - 追加カードは末尾に追加（既存の並びを崩さない）
- 文言は改変しない（改行・句読点の微調整のみ可）
- 変更差分は最小限。変更箇所が分かるようコメントを付与

# 作業手順
1) コードベース内で testimonial / testimonials / voice / review / customers などで検索し、
   お客様の声の「データ定義（配列）」または「CMS/JSON/markdown」など管理場所を特定する
   例：
   - /data/testimonials.ts
   - /constants/testimonials.js
   - CMSのJSON
   - markdownファイル
   - DBやAPIで取得している場合はseedや初期データ

2) 既存のカード表示ロジックは変更せず、データに1件だけ追加する
   - 既存データの1件をコピーし、同じフィールド構造のまま値を今回の事例内容に差し替える
   - id/slugが必要なら既存ルールに従い重複しない値を付与する
   - avatar画像が必須なら、既存のダミー画像/プレースホルダーを流用（新規画像追加は不要）
   - タグ/カテゴリが必須なら「AI記事制作」「SEO」「コンテンツ」などを既存仕様に合わせて設定

3) 追加後、PC/Tablet/SPで表示崩れがないことを確認（カード数が増えても崩れないこと）

# 追加する「お客様の声」コンテンツ（このまま使用）
【事例タイトル】
記事制作に追われなくなり、SEO施策を“継続できる状態”になりました

【属性】
業種：BtoBサービス（中小企業）
従業員数：20名／マーケ担当：1名（兼務）

【本文（カードFMTに合わせて格納）】
■ 導入前の悩み
ブログやコンテンツの重要性は理解していましたが、
日々の業務に追われ、記事制作の時間を確保できていませんでした。
SEOを意識した記事を書こうとしても、
「何を書けばいいのか」「この内容で正しいのか」が分からず、
更新が止まってしまうことも多くありました。

■ このAI記事制作サービスを選んだ理由
単に文章を自動生成するのではなく、
SEOを目的とした構成・キーワード設計まで含めて考えられていた点に魅力を感じました。
人がゼロから書く負担を減らしつつ、質は落とさない仕組みだと感じたのが決め手です。

■ 実施した内容
- 対策キーワード・記事テーマの整理
- SEOを意識した記事構成案の設計
- AIを活用した記事原稿の作成
- 人の確認・調整を前提とした運用フロー構築
- 継続更新を前提とした制作スケジュール設計
「完璧な記事を1本作る」より、“質を保ったまま継続できる状態”を目指して設計してもらいました。

■ 得られた成果・変化
- 記事制作にかかる時間が大幅に削減
- 更新頻度が安定し、コンテンツ施策が継続可能に
- SEOを意識した記事を量産できるようになった
- 徐々に検索流入が増え、コンテンツが資産として蓄積
特に大きかったのは、SEO施策を「気合い」ではなく「仕組み」で回せるようになったことです。

■ これから検討する方へ
コンテンツマーケティングは続けることが何より重要だと実感しました。
時間やリソースに限りがある企業ほど、AIを活用した記事制作は非常に相性が良いと思います。

# 既存カードFMTへのマッピング指示（必ず守る）
- もしカードが「短文コメント + 氏名/会社名/役職」のみの場合：
  - title = 上記事例タイトル
  - comment = 上記本文を1本にまとめて格納（見出し含めてOK、ただしUIを増やさない）
  - name/company/role が必須なら架空で自然な表記にする
    例：株式会社◯◯ マーケティング担当 / ◯◯様
- もし cards が sections や bullets の配列をサポートするなら、
  実施した内容/成果・変化は配列として格納し、表示仕様は既存に合わせる
- 表示項目（ラベルや見出し）を追加しない。既存UIに流し込むだけ。

# 納品要件（必ず出す）
- 変更したファイル名とパス
- どの配列/データに1件追加したか
- id/slug等をどう設定したか（重複なし）
- 既存UIが崩れていない確認結果（簡潔でOK）
- 可能な限り最小差分で変更し、差分が追いやすい状態にする
