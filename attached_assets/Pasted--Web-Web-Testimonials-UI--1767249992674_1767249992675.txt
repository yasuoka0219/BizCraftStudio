あなたは熟練のフルスタックエンジニア兼Webデザイナーです。
現在のWebサイトには「お客様の声（Testimonials）」のページ/セクション/カードUIが既に存在します。
今回は新規追加ではなく、既存の「お客様の声」カード（またはデータ）を、下記のMAツール事例に“差し替え”してください。

# 目的
既存Testimonialsの表示構造・デザイン・CSS・コンポーネントを一切崩さず、
表示されるテキスト内容のみをMAツールの事例に差し替える。

# 絶対条件（重要）
- レイアウト/デザイン/コンポーネント構造は維持（UIはそのまま）
- 既存のCSS/クラス名/余白/フォント/色/角丸/影などを変更しない
- 既存カードが複数ある場合：
  - まず「Testimonialsがどこで管理されているか（データ/props/CMS/JSON/markdown等）」を特定
  - 既存カードのうち「先頭の1件」または「指定ID/slugの1件」を本事例に置換
  - 既存の件数・並びは変えない
- 文言は改変しない（改行や句読点の微調整のみOK）
- TypeScript等の型定義があれば整合させ、型エラーを出さない
- 変更差分は最小限。変更した箇所が分かるようコメントを入れる

# 作業手順（やること）
1) コードベース内で testimonial / testimonials / voice / review / customers などで検索して、
   お客様の声の「データ定義」または「カードに流し込むprops」を管理している場所を特定する
   例：/data/testimonials.ts, /constants/testimonials.js, CMSのJSON, markdown など

2) 既存のカード表示ロジックはそのままに、該当データ（1件）を下記内容に差し替える
   - 既存が「title / subtitle / name / company / role / comment」形式なら、それに合わせてマッピングして入れる
   - 既存が「sections（見出し＋本文）」や「bullets（配列）」を持つなら、actions/results を配列で格納する
   - 既存が本文1つの長文フィールドのみなら、下記文章を見出し付きの体裁で1本に整形して入れる
   ※ただし表示枠や項目数は増やさない（UIを増やさない）

3) 実装後、プレビューで既存UIが崩れていないことを確認（レスポンシブ含む）

# 差し替える「お客様の声」コンテンツ（このまま使用）
【事例タイトル】
追いかけなくても、反応が返ってくる営業に変わりました

【属性】
業種：BtoBサービス（中小企業）
従業員数：22名／営業・マーケ担当：5名

【導入前の悩み】
リストや問い合わせ自体は一定数ありましたが、
「誰に・いつ・何を送るべきか」が完全に属人的でした。
対応が後回しになったり、
送る内容やタイミングが担当者ごとにバラバラになったりと、
せっかくの見込み顧客を活かしきれていない感覚がありました。
営業もマーケも、常に“追いかけ続ける状態”に疲れを感じていました。

【このMAツールを選んだ理由】
単にメールを自動送信するツールではなく、
顧客の行動や状態に合わせてアプローチを設計できる点に魅力を感じました。
また、既存の業務フローを無理に変えるのではなく、
「今の運用をベースに、どう自動化するか」を一緒に考えてくれたのも安心材料でした。

【実施した内容】（箇条書き）
- 見込み顧客の属性・行動パターンの整理
- 問い合わせ後・資料請求後など、シーン別シナリオ設計
- ステップメール／自動フォローの構築
- 営業が「今対応すべき顧客」だけ分かる仕組みづくり
- 運用開始後の改善（配信内容・タイミングの調整）
人が頑張り続ける前提ではなく、仕組みで成果が出る状態を目指して設計してもらいました。

【得られた成果・変化】（箇条書き）
- 手動フォローの工数が大幅に削減
- 反応のある見込み顧客が明確になり、営業の動きが効率化
- メール開封・返信率が向上
- 「今すぐ客」と「検討中客」を自然に分けられるようになった
特に大きかったのは、営業が“追いかける”のではなく、“反応を受け取る”形に変わったことです。

【これから検討する方へ】
MAツールは導入すれば成果が出るものではないと思っていましたが、
自社の顧客や動き方に合わせて設計することで、ここまで変わるとは思いませんでした。
営業やマーケの負荷を減らしながら成果を伸ばしたい方には、非常に相性の良い仕組みだと思います。

# 形式の注意（既存カード項目に合わせる）
- 既存が短文コメント＋氏名/会社名のみ必須なら：
  - title（見出し）＝上記タイトル
  - comment（本文）＝「導入前の悩み」〜「これから検討する方へ」を1本にまとめて入れる
  - name/company/role が必須なら、架空で自然な表記にする（例：株式会社◯◯ マーケ責任者 / ◯◯様）
  ※UIの項目数や表示枠は増やさない

# 納品要件（必ず出す）
- 変更したファイル名とパス
- どのデータ（何件目/どのID）を差し替えたか
- 既存UIが崩れていない確認結果（簡単でOK）
- 可能なら最小差分になるように変更し、差分が追いやすい形にする
